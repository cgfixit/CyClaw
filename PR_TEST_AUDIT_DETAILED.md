## Executive Summary

The CyClaw test suite is **GREEN and genuinely healthy** — 463 passed, 0 failed, 0 collection errors in ~27 s on Python 3.12.3 (HEAD `5795dfe`), RAG smoke 4/4 vault-hits, 88% total coverage (clears the 80% gate). **This is a quality-and-coverage hardening exercise, not a triage.**

The dominant strength: files exercise the **real code at the right seams** — the real FastAPI app via `TestClient` (graph mocked only at the `gate.compiled_graph` DI seam), the real compiled LangGraph, the real `HybridRetriever` RRF math (via an unbound-method + `SimpleNamespace` pattern that needs no on-disk index), the real `PersonalityManager` + SQLite, real subprocess argv builders with no-shell assertions. Security instincts are strong: the suite pins the triple-gate Grok routing, the soul injection gate, the BM25 pickle-rejection (JSON-only loader), API-key fail-closed auth, prompt-injection redaction, and the no-shell subprocess contract.

The real weaknesses **cluster into a handful of cross-cutting themes**, not per-file defects:

1. **Hermeticity leak (highest-value fix):** `test_graph.py` redirects `audit_file` into `tmp_path` and sets `SAFECLAW_CONFIG` — but **`SAFECLAW_CONFIG` is read by nothing**, and `audit_log()` resolves the hardcoded `config.yaml` and caches it. So every run appends to the repo's real `logs/audit.jsonl` (empirically confirmed 50→60 lines). Tests only assert on the returned `result['audit_event']` dict, so the dead redirection is invisible.
2. **False confidence:** two `test_gate.py` `GET /soul` success tests pass only by accident of the on-disk `config.yaml` (`personality.enabled=True`) + a real `soul.md`; the fixture's `patch('gate.cfg', cfg)` is dead for the personality path (`gate.personality` is bound once at import).
3. **CI gates an explicit hand-picked file list, not `pytest tests/`:** `test_ops_runner.py` runs locally (part of the 463) but is **absent from CI and from `--cov`** — so the argv-injection/architectural-isolation guarantees for two browser-reachable, write-capable `/ops` endpoints are unenforced by CI.
4. **Happy-path-only CLIs/selftests/schedulers:** the source of the 71–78% modules (`agentic/cli.py`, `sync/selftest.py`, `sync/scheduler.py`, `mcp_hybrid_server.py`, …) — error/exit-code branches the modules exist to produce are untested.
5. **Under-covered modules / two missing test files:** `personality_db.py` 67% (entire Postgres branch untested, no `test_personality_db.py`), `gate.py` 68% (429 / 500-redaction / soul-mutation / `/ops` HTTP paths), `errors.py` 96% indirect (no `test_errors.py`).
6. **A few vacuous assertions** (notably `test_stemmer.py`) + maintainability noise (dual mock convention in conftest, duplicated `TEST_CONFIG`, script-style `test_personality_changes.py`, stale "verified against HEAD …" banners, a suite-wide `StarletteDeprecationWarning`).

**Suite health score:** **B+** — strong, green, security-aware, with quality gaps in hermeticity, false-confidence, CI gating, and breadth. Fix the audit leak, the soul-test dependency, and the CI gating and this is solidly **A-**.

## Cross-Cutting Themes (root causes — fix these, not 38 files)

### 1. Non-hermetic audit writes / dead config-redirection plumbing

test_graph.py's _make_cfg redirects audit_file into tmp_path and sets os.environ.setdefault('SAFECLAW_CONFIG', ...), but SAFECLAW_CONFIG is read by NOTHING in the codebase. utils.logger.audit_log() resolves config via the hardcoded 'config.yaml' and caches it, so reset_config_cache() just reloads the real repo config whose audit_file is 'logs/audit.jsonl'. Empirically confirmed: running test_graph.py appends 10 lines to the real logs/audit.jsonl (50->60). Tests pass only because they assert on the returned result['audit_event'] dict, never on file contents, so the redirection's failure is invisible. The os.environ.setdefault also leaks a junk env var across the session. test_security.py's TestLoggingSetup similarly reaches into private logger state and clears the cyclaw root logger's handlers (a global side effect).

### 2. False confidence: tests passing by accident of on-disk repo state, not fixtures

gate.personality is bound ONCE at import from the real config.yaml (personality.enabled=True today), so test_get_soul_accepts_correct_key and test_get_soul_audit_logged return 200 only because of the repo's on-disk config + data/personality/soul.md; the fixture's patch('gate.cfg', cfg) is dead for the personality path. If config.yaml flips personality.enabled=false or soul.md is removed, both tests flip to 404 and fail despite gate.py being correct. Related: test_agentic_cli.py points the registry at the REAL committed data/agentic/skills_registry.json (no per-test isolation), safe only because no current test writes it; test_agentic_selftest.py's 'all pass without gh' test depends on gh being absent from PATH (its pass/fail is a property of the runner's machine); test_rag_integration.py calls bare build_index() writing the shared on-disk index dirs.

### 3. CI gates an explicit hand-picked file list, leaving safety-critical tests ungated

CI runs an explicit 35-file list, NOT 'pytest tests/'. test_ops_runner.py is present and passes locally (part of the 463) but is absent from the CI list, and utils.ops_runner is absent from the --cov targets. ops_runner is the subprocess boundary for POST /ops/sync and POST /ops/agentic (browser-reachable, write-capable) and carries the argv-injection-safety and architectural-isolation guarantees. A regression (body inlined into argv, or sync/agentic imported into the gateway) would pass CI green. New test files created during remediation (test_personality_db.py, test_errors.py) would also be ungated unless explicitly added.

### 4. Happy-path-only CLIs / selftests / schedulers leave error and exit-code branches untested

This is the single biggest coverage-shape theme and the source of the 71-78% modules. agentic/cli.py: cmd_context enabled path, apply-skill success + injection-block, cmd_test, --body-file all untested. sync/cli.py: every cmd_setup error branch, cmd_sync SyncError-raise path, cmd_status scheduler-state branches, cmd_unschedule branches. sync/selftest.py & agentic/selftest.py: every FAIL/SKIP-alternate branch of the per-check loop. sync/scheduler.py: entire WindowsTaskScheduler.status() plus all subprocess error branches. mcp_hybrid_server.py: unknown-tool -32601, the whole main() stdio loop, the -32700 parse-error path. agentic/writer.py: the pr_create op (entire 45-49 coverage hole). ops_runner: subprocess-raise cleanup, propose-skill happy path, no-body branch. Pattern: tests drive the all-green path and one or two refusals, but the modules exist to surface failures loudly.

### 5. Two mock conventions in conftest, only one of which is guarded

conftest.py ships TWO parallel mock conventions: fixture-style (test_config, mock_retriever, mock_llm, mock_search_results, bm25_index) and class-style (MockRetriever/MockLocalLLM/MockGrokClient + MOCK_*_RESULTS constants). test_conftest_fixtures.py guards only the class-style side. None of the five @fixtures are exercised, and MockGrokClient.is_available() — the security-relevant flag that drives graph routing in test_graph.py:158 and the triple-gate — is imported but never tested. If is_available()'s available= kwarg wiring drifts, the meta-test stays green while the routing tests silently lose their teeth.

### 6. Lowest-coverage modules and two missing dedicated test files

personality_db.py is 67% — its entire Postgres branch (connect() ImportError + success, both Postgres DDL strings) is untested and there is NO tests/test_personality_db.py. The module is pure/dependency-light and trivially testable; the only mock needed is a fake psycopg in sys.modules. errors.py is 96% via indirect use but has no dedicated test; three subclass __init__ code-strings (EMBEDDING_ERROR/INDEX_NOT_FOUND/CORPUS_EMPTY) are uncovered and the fragile Sync* two-step code-override pattern is unpinned. gate.py (68%) is missing /query 429, /query 500 graph_error redaction, soul mutation endpoints, and /ops endpoints at the HTTP level. mcp_hybrid_server.py (72%) is missing main()/unknown-tool/parse-error.

### 7. Vacuous / accidentally-passing assertions and stale documentation banners

test_stemmer.py has three weak assertions: test_filters_short_words asserts len(tokens)>=1 (passes for any non-empty input and contradicts its own name); test_numeric_ignored asserts all(t.isalpha()) which is FALSE in general (k8s, a1 survive); test_basic_tokenization asserts a magic count==4 without naming tokens. test_startup_robustness.py captures (host,port) but asserts only len(served)==1, so a 0.0.0.0 loopback-binding violation would pass. Stale banners: test_audit.py header claims '7 tests' (actually 13); test_personality_changes.py and test_personality.py pin 'verified against HEAD f5934db'; test_gate.py has a 'REVIEW THIS SOON' docstring + unused MockGrokClient import. test_personality_changes.py is script-style (__main__ runner, 6 prints, sys.path.insert) and duplicates 3 of 4 sibling tests.

### 8. Security invariants asserted at node/unit level but not end-to-end through the graph

Audit Convergence (invariant 4) is directly asserted for only 2 of 6 graph paths; the grok path and the user_gate pause path never assert audit_event presence/online_escalated/model_used. The personality soul-injection in local_llm_node (graph.py:194) and record_interaction in audit_logger_node (graph.py:389-404, incl. the non-fatal except) are entirely untested — the privacy-relevant 'only the query hash is stored' and 'DB failure never breaks query flow' contracts are unverified. Separately, NO test runs the real HybridRetriever through the real graph: test_graph.py uses a MockRetriever and test_rag_integration.py stops before the graph despite its 'integration' name.

### 9. Duplication and redundant expensive operations across test files

test_personality.py defines its own TEST_CONFIG (line 18) that is fully overwritten by the cfg fixture before use (dead, plus a shallow-copy aliasing hazard) and duplicates conftest.TEST_CONFIG. test_personality_changes.py duplicates 3 of its 4 tests against the sibling. test_rag_integration.py is a near-exact duplicate of ci_rag_smoke.py's first two queries running in the same CI job and re-running a full (expensive) build_index over real embeddings for no marginal signal. ~12 test_personality.py tests repeat an identical 4-line patched-import preamble.

### 10. Suite-wide StarletteDeprecationWarning from FastAPI TestClient

The single suite warning is StarletteDeprecationWarning: 'Using httpx with starlette.testclient is deprecated; install httpx2 instead', emitted by every TestClient-based test file (test_gate.py, test_security.py via gate.app). It is non-fatal today but will become actionable if the project adds -W error, and signals an upcoming httpx/starlette test-client migration. Files that use httpx.Client directly (test_client.py) or no TestClient are insulated.

### `test_gate.py`  —  **minor-issues**

> Solid, real-code integration tests for the happy paths and auth/security surface, but several high-value error/security wiring paths (graph_error 500 redaction, rate-limit 429, soul mutation endpoints, ops endpoints) are untested, and two soul tests silently depend on the real config.yaml + on-disk soul file rather than the fixture.

- **[HIGH] Two GET /soul success tests silently depend on the real config.yaml and on-disk soul.md, not the fixture**  _(false-confidence)_
    - **What:** test_get_soul_accepts_correct_key (lines 249-260) and test_get_soul_audit_logged (lines 262-277) expect HTTP 200 from GET /soul, which requires gate.personality to be a live PersonalityManager. But the conftest TEST_CONFIG sets personality.enabled=False, and the fixture only does patch('gate.cfg', cfg) / sets gate.cfg=cfg. gate.personality is constructed ONCE at module import (gate.py line 261-263) from the REAL config.yaml on disk, which currently has personality.enabled: True and a real soul file at data/personality/soul.md. I verified this directly: importing gate yields gate.personality = PersonalityManager(soul_path='data/personality/soul.md'). So these two tests pass only by accident of the repo's on-disk config and soul file. If config.yaml is flipped to personality.enabled: false (a legitimate deployment), or soul.md is absent, both tests flip to 404 and fail — even though gate.py is correct. The patch('gate.cfg', cfg) in the fixture is dead/misleading for the personality path because personality was already bound at import.
    - **Fix:** Make the personality dependency explicit and hermetic: in these two tests (or the fixture), set gate.personality to a MagicMock whose get_system_prompt_additive()/get_version()/soul_path return fixed values, inside a try/finally that restores the original (mirror the existing test_get_soul_404_when_disabled pattern that already sets gate.personality = None). That removes the hidden dependency on config.yaml and data/personality/soul.md and makes the assertion test gate.py's behavior rather than the repo's filesystem state.
- **[HIGH] graph_error 500 path and its _sanitize_error wiring (lines 306-311) are untested at the HTTP level**  _(coverage-gap)_
    - **What:** When compiled_graph.invoke raises, gate.py catches it, calls _sanitize_error(e), writes a graph_error audit event, and returns HTTP 500 with detail.code='GRAPH_ERROR' (lines 306-311). This is the security-critical integration: a real graph crash whose message contains a credential must NOT leak it in the 500 body. _sanitize_error is unit-tested in isolation (TestErrorSanitization), but the wiring — that the /query endpoint actually routes a raised exception through _sanitize_error before responding — has zero coverage. A regression that removed the _sanitize_error call (returning str(e) directly) would not be caught by any test.
    - **Fix:** Add a test that sets mock_graph.invoke.side_effect = RuntimeError('boom GROK_API_KEY=<secret>') with that secret set via monkeypatch.setenv, POSTs /query, asserts resp.status_code == 500, resp.json()['detail']['code'] == 'GRAPH_ERROR', and the secret value is NOT present anywhere in resp.text. Also assert a graph_error audit event was written to the tmp audit file.
- **[HIGH] /query rate-limit 429 path (lines 274-279) is never exercised through the HTTP layer**  _(coverage-gap)_
    - **What:** check_rate_limit gating /query (lines 273-279) — emit rate_limit_exceeded audit event and raise HTTP 429 with code='RATE_LIMIT' — has no integration test. test_rate_limit.py only calls gate.check_rate_limit() directly and asserts gate._rate_limiter is the shared instance; it never drives the /query endpoint to a 429. The HTTP error envelope shape (detail.error / detail.code) and the audit emission are unverified.
    - **Fix:** Add a test that patches gate.check_rate_limit to return False (cleanest, avoids hammering the shared limiter), POSTs /query, asserts 429 and resp.json()['detail']['code'] == 'RATE_LIMIT', and that a rate_limit_exceeded event landed in the tmp audit file. Patching the function rather than exhausting the real limiter also keeps the test from leaking counter state into the module-level _rate_limiter (see related flakiness finding).
- **[MEDIUM] Soul mutation endpoints /soul/propose, /soul/apply, /soul/restore (lines 371-408) have no integration tests**  _(coverage-gap)_
    - **What:** Only GET /soul and POST /soul/reload (404-disabled branch) are tested. The mutating endpoints are untested at the HTTP level: propose_soul_evolution (371-377), apply_soul_evolution including the PromptInjectionError -> 400 branch with soul_apply_injection_blocked audit (379-391), and restore_soul including the FileNotFoundError -> 404 branch (400-408). These are the soul-governance security surface (invariant #5) — the injection gate on apply and the require_api_key gating are exactly the behaviors most worth pinning. ## Test-Writing Playbook (conventions derived from the strongest files)

CyClaw test-writing conventions, derived from the strongest files in the suite (test_rate_limit.py, test_client.py, test_hybrid_search.py, test_agentic_context.py, test_agentic_isolation.py). Follow these when hardening the under-covered modules.

1) TEST THE REAL CODE AT THE RIGHT SEAM. Import and call the production symbol; mock only the genuine I/O boundary. Examples that are already idiomatic: patch httpx.Client.post but construct real httpx.Request/Response so raise_for_status/json run for real (test_client.py); patch gate.compiled_graph (the DI seam) but drive the real FastAPI app via TestClient (test_gate.py); patch context.run_read but run the real fetch_* bundling (test_agentic_context.py). NEVER reimplement the logic under test in the test.

2) UNBOUND-METHOD + SimpleNamespace('self') FOR INDEX-REQUIRING CLASSES. To test HybridRetriever math without an on-disk ChromaDB/BM25 index, call HybridRetriever.hybrid_search / _normalize_single_path as unbound methods with a SimpleNamespace supplying only the attributes the method reads (rrf_k, semantic_search, keyword_search, top_k_*). This runs the production method body with zero index. Reuse this for keyword_search by injecting a fake self.bm25 with get_scores returning known positive/zero scores.

3) HERMETIC FIXTURES — tmp_path FOR ALL I/O, NEVER /tmp OR REPO PATHS. Write config.yaml, audit/log files, ChromaDB/BM25 paths, registry files, and soul.md under tmp_path. Two repo-specific gotchas: (a) audit_log() resolves the HARDCODED 'config.yaml' and caches globally, so redirecting audit output requires monkeypatching utils.logger._config_cache (or _get_config) after reset_config_cache(), NOT setting an env var — the SAFECLAW_CONFIG var is read by nothing. (b) AgenticConfig/SyncConfig validators REQUIRE registry_path under data/ and local_path under data/corpus, so for those use a unique path like f'data/agentic/_pytest_{uuid4().hex}.json' with a teardown unlink (see test_agentic_registry.py), not tmp_path.

5) FAKE CLOCK / INJECTED SEAMS INSTEAD OF sleep. For time-dependent logic inject a clock= callable (RateLimiter pattern) so window expiry/eviction/stale-lock reclaim are deterministic — back-date mtimes with os.utime(path,(old,old)) rather than waiting. The only justified real sleep is a sub-millisecond GIL-yield to expose a missing lock (test_rate_limit.py _SlowHits), paired with an EXACT-equality assertion.
